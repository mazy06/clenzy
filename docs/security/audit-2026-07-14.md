# Audit de sécurité applicatif — Baitly (PMS location courte durée)

- **Date** : 2026-07-14
- **Périmètre** : dépôt `clenzy` (backend `com.clenzy.*` Java 21 / Spring Boot 3.5.13, frontend React 18 / MUI, mobile Expo/React Native, SDK booking, `nginx.prod.conf`).
- **Type** : audit défensif, **lecture seule** (aucune modification de code).
- **Méthode** : 10 phases (reconnaissance → frontend), findings prouvés par `fichier:ligne` et vérifiés manuellement.
- **Hors périmètre** (dépôts séparés, non auditables ici) : `clenzy-infra` (docker-compose prod, ACL bucket OVH/MinIO, ACL Kafka, IAM), `clenzy-sites` (SSR des sites booking publiés).

> **Note de cadrage** : le brief annonçait Airwallex + AWS. La réalité du code est **Stripe (primaire) + PSP marocains (CMI/Attijari, PayTabs, Payzone)** et **OVH/MinIO + Docker/nginx**. L'audit porte sur le stack réel.

---

## 1. Résumé exécutif

La plateforme présente une **base de sécurité mature** : deny-by-default, validation JWT durcie (issuer + audience + signature), chiffrement PII au repos, signatures webhook de paiement correctes, montants recalculés serveur, rate limiting robuste, gestion DLT/poison Kafka, aucun secret de prod en dur ni dans l'historique git, frontend sans sink XSS. Les campagnes d'audit antérieures (2026-06) ont clairement durci le code.

**Mais** l'isolation multi-tenant — risque n°1 d'un PMS SaaS — repose sur une **architecture fragile sans filet** : **pas de RLS PostgreSQL** et le **filtre Hibernate `organizationFilter` est inerte sur les flux HTTP**. L'isolation dépend donc d'une vérification d'ownership explicite dans **chaque** endpoint. Plusieurs oublis produisent des **IDOR cross-tenant confirmés**, dont un critique.

### Top 5 des risques

| # | ID | Risque | Sévérité |
|---|-----|--------|----------|
| 1 | **F1-01** | Tout utilisateur authentifié lit **n'importe quelle réservation** (PII guest, montants, statut paiement) par simple incrément d'ID | **CRITICAL** (7.7) |
| 2 | **F3-01** | **SSRF non authentifiée** avec exfiltration (snapshot Playwright + sous-ressources CSS non validées) | **HIGH** (8.6) |
| 3 | **F1-03** | Tout utilisateur authentifié **initie le paiement / mute** la facture d'un autre tenant | **HIGH** (8.1) |
| 4 | **F1-STRUCT** | **Pas de RLS + filtre Hibernate inerte** → aucune seconde ligne de défense ; amplifie tous les IDOR | **HIGH** (7.4) |
| 5 | **F1-02** | Énumération cross-tenant de tout l'historique documentaire (emails, n° de factures) par un HOST | **HIGH** (7.1) |

### Verdict : **NO-GO en l'état pour une mise en production commerciale grand public**

Le cluster d'IDOR cross-tenant (**F1-01 → F1-09**) et la **SSRF anonyme (F3-01)** sont des **bloquants** : une fuite de données entre tenants et une SSRF non authentifiée sont incompatibles avec une commercialisation SaaS. Ce sont majoritairement des **quick wins** (ajout d'une garde d'ownership déjà standardisée dans le code via `OrganizationAccessGuard`).

**Go conditionnel** dès que : (a) les IDOR F1-01→F1-09 sont gardés, (b) F3-01 est corrigé (auth + IP épinglée), (c) le déficit structurel F1-STRUCT reçoit un filet (RLS ou réactivation du filtre par transaction). Les autres findings (Medium/Low) peuvent suivre en court terme.

## 3. Détail des findings prioritaires

### F1-STRUCT — Isolation 100 % applicative, sans filet (High, CVSS 7.4)

**Cause racine de tout le cluster IDOR.** Deux mécanismes de défense théoriques sont absents/inertes :

1. **Aucune RLS PostgreSQL** (aucune `CREATE POLICY` / `ENABLE ROW LEVEL SECURITY` dans les changesets Liquibase).
2. **Filtre Hibernate `organizationFilter` inerte sur les flux HTTP** : `open-in-view: false` sur les 4 profils, et **aucun aspect/intercepteur ne réactive le filtre par transaction**. Les 3 seuls `enableFilter` sont hors flux HTTP transactionnel — `TenantFilter.java:300-309` échoue faute d'EntityManager transactionnel (aveu dans son propre Javadoc), `TenantScopedExecutor` (async), `DocumentPreviewService` (explicite).

```java
// TenantFilter.java:300 — le filtre échoue silencieusement au niveau servlet
private void enableHibernateOrgFilter(Long orgId) {
    try {
        Session session = entityManager.unwrap(Session.class); // IllegalStateException (open-in-view=false)
        session.enableFilter("organizationFilter").setParameter("orgId", orgId);
    } catch (IllegalStateException e) {
        logger.debug("TenantFilter: filtre Hibernate non activable ..."); // avalé
    }
}
```

**Conséquence** : chaque `findById` en flux HTTP dépend d'une garde explicite `requireSameOrganization`. Toute garde oubliée = fuite cross-tenant totale, sans seconde ligne de défense.

**Correctif** (au choix, idéalement les deux) :
- **RLS Postgres** : `ALTER TABLE ... ENABLE ROW LEVEL SECURITY` + policy `USING (organization_id = current_setting('app.current_org')::bigint)`, la var de session posée par le `TenantFilter`.
- **Réactiver le filtre par transaction** : un aspect `@Around` sur `@Transactional` (ou un `TaskDecorator`/`Interceptor` Hibernate) qui `enableFilter` sur la session courante à partir du `TenantContext`.

### F1-01 — IDOR lecture `GET /api/reservations/{id}` (Critical, CVSS 7.7)

`AV:N/AC:L/PR:L/UI:N/S:C/C:H/I:N/A:N`

```java
// ReservationController.java:131 — classe @PreAuthorize("isAuthenticated()") seulement
@GetMapping("/{id}")
public ResponseEntity<ReservationDto> getById(@PathVariable Long id) {
    Reservation reservation = reservationService.getByIdFetchAll(id); // findByIdFetchAll: WHERE r.id = :id (aucun org)
    return ResponseEntity.ok(reservationMapper.toDto(reservation));   // aucun validatePropertyAccess
}
```

**Exploitabilité** : n'importe quel utilisateur authentifié (même HOUSEKEEPER/TECHNICIAN) itère `id` et lit toute réservation de tout tenant : PII guest, dates, montants, statut de paiement, codes d'accès. Les endpoints voisins (`update`, `cancel`, `sendPaymentLink`) appellent bien `validatePropertyAccess(...)` — **seul `getById` a été oublié**.

**Correctif** :
```java
@GetMapping("/{id}")
public ResponseEntity<ReservationDto> getById(@PathVariable Long id, @AuthenticationPrincipal Jwt jwt) {
    Reservation reservation = reservationService.getByIdFetchAll(id);
    validatePropertyAccess(reservation.getProperty().getId(), jwt); // garde déjà utilisée par les autres endpoints
    return ResponseEntity.ok(reservationMapper.toDto(reservation));
}
```

### F1-03 — IDOR écriture `POST /api/invoices/{id}/pay` (High, CVSS 8.1)

`AV:N/AC:L/PR:L/UI:N/S:C/C:L/I:H/A:N`

```java
// InvoicePaymentService.java:39 — aucun contrôle d'org
public PaymentOrchestrationResult payInvoice(Long invoiceId, ...) {
    Invoice invoice = invoiceRepository.findById(invoiceId).orElseThrow(...); // pas de requireSameOrganization
    ...
    invoice.setPaymentTransactionId(result.transaction().getId()); // mutation cross-tenant
    invoice.setPaymentMethod(result.providerUsed().name());
    invoiceRepository.save(invoice);
}
```

Tout utilisateur authentifié initie l'orchestration de paiement et mute la facture d'un autre tenant. `sendInvoice` (F1-04) a le même défaut (DRAFT→SENT). Dans le **même** controller, `getInvoice`/`pdf`/`issue`/`cancel` sont org-validés — `payInvoice`/`sendInvoice` sont les deux seules méthodes non gardées.

**Correctif** : charger la facture, puis `organizationAccessGuard.requireSameOrganization(invoice.getOrganizationId(), "...")` avant toute action ; ou requête `findByIdAndOrganizationId`.

### F1-02 — Énumération cross-tenant de l'historique documentaire (High, CVSS 7.1)

```java
// DocumentController.java:238
@GetMapping("/generations")
@PreAuthorize("hasAnyRole('SUPER_ADMIN','SUPER_MANAGER','ADMIN','HOST')") // HOST = tenant-scoped
public ResponseEntity<Page<DocumentGenerationDto>> listGenerations(...) {
    return ResponseEntity.ok(generatorService.listGenerations(pageable));
}
// DocumentGeneratorService.java:261 → findAllByOrderByCreatedAtDesc(pageable) : AUCUN filtre org
```

Un HOST pagine l'historique de génération documentaire de **tous les tenants** (emails destinataires, numéros légaux de factures, refType/refId) sans deviner d'ID. L'endpoint voisin `getGenerationsByReference` post-filtre par email pour les non-staff — oubli isolé sur `listGenerations`.

**Correctif** : `findAllByOrganizationIdOrderByCreatedAtDesc(TenantContext.orgId, pageable)`.

### F3-01 — SSRF non authentifiée + exfiltration (High, CVSS 8.6)

`AV:N/AC:L/PR:N/UI:N/S:C/C:H/I:L/A:N`

```java
// SitePreviewProxyController.java:149 — /api/public/preview-proxy/snapshot?url= (ANONYME)
@GetMapping("/snapshot")
public ResponseEntity<String> snapshotSite(@RequestParam String url) { ... snapshotService.captureSnapshot(url); }

// SiteSnapshotService.java
private void validateUrl(String url) { ICalUrlValidator.validateAndResolve(url); } // IP épinglée JETÉE
page.navigate(url, ...);                       // l.132 — suit les 3xx + re-résout le DNS (rebinding)
String cssText = Jsoup.connect(href).execute().body(); // l.263 — href de la page attaquante, AUCUNE validation
link.after("<style>" + cssText + "</style>");  // CSS interne INLINÉ dans la réponse → exfiltration
```

**Exploitabilité** : anonyme. Une URL publique validée qui redirige vers `169.254.169.254` / une IP interne est rendue par Chromium ; et les `<link rel=stylesheet>` de la page attaquante permettent de fetch n'importe quelle URL interne et d'**inliner la réponse** dans le HTML retourné (metadata cloud, services internes MinIO/actuator/go2rtc).

**Correctif** : (a) retirer l'accès anonyme (auth + rate-limit), (b) intercepter chaque requête Playwright via un `Route` qui revalide l'IP, (c) valider les `href` CSS via `ICalUrlValidator` **et** router le fetch via le pattern IP-épinglée (`PinnedSiteFetcher`).

### F4-01 — Webhook Wise fail-open sur signature (Medium maintenant / High à l'activation)

```java
// WiseWebhookController.java:75
if (wisePublicKeyPem != null && !wisePublicKeyPem.isBlank()) {
    ... vérifie la signature RSA-SHA256, rejette si invalide ...
} else {
    log.warn("... vérification de signature désactivée (wise.public-key vide) ..."); // FAIL-OPEN
}
```

`wise.public-key` est **absente d'`application-prod.yml`** (défaut `@Value("${wise.public-key:}")` = vide). Un payload non signé forgé passerait un payout en `PAID`/`FAILED`. **Actuellement neutralisé** car `/api/webhooks/payouts/**` n'est pas dans le `permitAll` (403). Devient exploitable dès qu'on whiteliste l'endpoint pour activer Wise. Contraste : GoCardless **rejette (503)** si le secret est absent.

**Correctif** : fail-closed — rejeter si `wise.public-key` absente, comme GoCardless/Stripe.

### F6-01 — Broker Kafka `PLAINTEXT` par défaut (Medium, CVSS 6.5)

```yaml
# application-prod.yml:72
security.protocol: ${KAFKA_SECURITY_PROTOCOL:PLAINTEXT}   # SASL/SSL câblés mais désactivés par défaut
sasl.jaas.config: ${KAFKA_SASL_JAAS_CONFIG:}              # vide
```

Sans `KAFKA_SECURITY_PROTOCOL=SASL_SSL` en prod : aucune authentification, aucun chiffrement, aucune ACL → toute partie avec accès réseau au broker peut produire/consommer tous les topics. Cela rend exploitables les gaps de confiance des consumers (**F6-02** : forge d'un event `documents.generate` → document d'un autre tenant envoyé à une adresse arbitraire ; double-effet `PaymentEventConsumer`). Sévérité finale conditionnée à l'isolation réseau du broker (**à confirmer dans `clenzy-infra`**).

**Correctif** : `SASL_SSL` + SCRAM-SHA-512 + truststore + **ACL par topic/consumer-group** ; enrober chaque `@KafkaListener` touchant la DB dans `TenantScopedExecutor.runAsOrganization(orgId)` après dérivation de l'orgId depuis une source de confiance.

## 4. Plan de remédiation priorisé

### Quick wins (< 1 jour) — **bloquants GA**
1. **F1-01 → F1-09** : ajouter la garde `organizationAccessGuard.requireSameOrganization(entity.getOrganizationId(), ...)` (déjà standardisée) sur chaque endpoint IDOR : réservations `getById`, invoices `pay`/`send`, cluster ServiceRequest, documents `generations`/`by-reference`. → requêtes `findByIdAndOrganizationId` là où pertinent.
2. **F3-01** : retirer l'accès anonyme au snapshot + IP épinglée sur page ET sous-ressources CSS.
3. **F4-01** : rendre la vérif de signature Wise obligatoire (fail-closed) **avant** toute activation de Wise.

### Court terme (< 1 semaine)
4. **F1-STRUCT** : introduire un filet — RLS Postgres **ou** réactivation du filtre Hibernate par transaction (aspect). *Le changement le plus structurant : sans lui, chaque nouvel endpoint reste un IDOR potentiel.*
5. **F6-01/F6-02** : activer SASL_SSL + ACL Kafka ; enrober les consumers via `TenantScopedExecutor` + revalider l'appartenance des `referenceId`/escrow↔réservation.
6. **F3-02/03/04** : router les fetch/webhooks sortants via le pattern IP-épinglée ; allow-list schéma/hôte pour les sources caméra.
7. **F3-06/07** : élargir la liste non-auto-approuvable IA aux tools de communication/annulation ; reclasser `remember_fact` en écriture tracée/validée.
8. **F7-01** : rendre le gate SCA bloquant (retirer `|| true`/`continue-on-error`, `failBuildOnCVSS=7`, vrai fichier de suppressions) ; `npm audit fix`.
9. **F4-02/03** : contrainte unique `idempotency_key` + UPDATE conditionnel ; index unique `stripe_session_id`.

### Structurel / conformité
10. **F8-01/02** : activer la purge de rétention en prod (durées légales par catégorie) ; étendre l'effacement/export aux `Guest`/`GuestDeclaration`.
11. **F8-03** : confirmer DPA/SCCs avec les sous-traitants LLM US (Anthropic/OpenAI/Voyage) ; minimiser la PII dans les prompts.
12. **F5-01** : révoquer/roter les secrets **test** commités, externaliser (`.env.local` non commité).
13. **F5-02 / F5-03 / F2-* / F9-01 / F6-03** : durcir CSP (nonces, sans `unsafe-*`) ; confirmer `proxy_ssl_verify` ; allow-list rôle sur templates WhatsApp + mapping de rôle strict + révocation au logout ; allow-list des URLs de retour paiement ; masquage PII systématique en logs.

## 5. Points positifs (à préserver)

- **Deny-by-default** : `SecurityConfigProd` termine par `.anyRequest().denyAll()` ; profils de sécurité en liste positive + `EnvironmentValidator` fail-fast (pas de `admin/admin`, `JASYPT_ENCRYPTOR_PASSWORD` requis).
- **Validation JWT durcie** : issuer + **audience** (`clenzy-api`) + signature JWKS + timestamps (`ProdJwtDecoderConfig`/`JwtAudienceValidator`) → rejette cross-realm ET cross-client.
- **Résolution tenant fail-closed**, dérivée **du JWT seul** (`TenantFilter`), `TenantContext.clear()` en `finally` ; garde centralisée `OrganizationAccessGuard` (fail-closed) largement utilisée.
- **Paiements** : signatures webhook correctes (Stripe/CMI/Attijari/PayTabs/Payzone), montants **recalculés serveur** (cross-check 400), `StripeAmounts.toMinorUnits` (HALF_UP), transitions gardées (CAS), numérotation factures concurrente-safe, secrets chiffrés per-org, `StripeGateway` (RequestOptions par appel), **aucun log de PAN/CVV/secret**.
- **Pas d'injection SQL** (requêtes paramétrées) ; **import iCal SSRF-safe** (IP épinglée) ; **scoping tenant des agents IA solide** (le LLM ne fournit jamais l'org ; RAG scopé org ; tools argent jamais auto-approuvés ; HITL structurel).
- **Rate limiting robuste** (Lua atomique, brute-force auth/voucher/key-verify, résolveur d'IP anti-spoofing) ; **uploads** validés (MIME + taille + pas de path traversal) ; **DLT/poison Kafka** robustes.
- **Cookies BFF** HttpOnly/Secure/SameSite=Strict (refresh scopé) ; **frontend sans sink XSS** (HTML non fiable sandboxé) ; **tokens mobile en SecureStore** ; **PII chiffrée au repos** (`Guest`, `GuestDeclaration`, `User`…) ; en-têtes/TLS nginx solides ; **aucun secret de prod en dur ni dans l'historique git**.
