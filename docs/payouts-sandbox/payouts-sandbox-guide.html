1. Vue d'ensemble & architecture

Ce guide accompagne le déploiement local des deux nouveaux rails de payouts sortants automatisés de Clenzy : Wise Business pour les virements internationaux (Maroc, Arabie Saoudite, 80+ pays) et GoCardless Bank Account Data + PIS pour les virements SEPA automatiques sans upload manuel.

Les 5 méthodes de payout supportées par Clenzy

MéthodeCouvertureNiveau d'autoFrais typiques
STRIPE_CONNECTEU, US, UK, +40 paysAuto complet0.25% + €0.10
WISE80+ pays (dont MA, KSA)Auto complet~0.5%
OPEN_BANKINGEU (PSD2)Auto via PIS~€0.10/virement
SEPA_TRANSFEREU (zone SEPA)Semi-auto (XML + upload manuel)Variable banque
MANUAL—Hors-ClenzyN/A

Architecture Strategy + Registry

Tous les exécuteurs implémentent l'interface {@code PayoutExecutor} et sont auto-découverts par Spring. L'orchestrateur {@code PayoutExecutionService} délègue à l'exécuteur via {@code PayoutExecutorRegistry.get(method)}.

com.clenzy.payment.payout/
├── PayoutExecutor              (interface)
├── PayoutExecutorRegistry      (auto-discovery Spring)
├── PayoutNotifier              (notifications partagées)
├── wise/WiseClient             (HTTP REST Wise API)
├── openbanking/
│   ├── GoCardlessPisClient     (HTTP REST GoCardless + cache token)
│   └── OpenBankingProvider     (enum GOCARDLESS / TINK)
└── executor/
    ├── StripeConnectPayoutExecutor
    ├── SepaTransferPayoutExecutor
    ├── ManualPayoutExecutor
    ├── WisePayoutExecutor
    └── OpenBankingPayoutExecutor

2. Setup ngrok (tunnel local)

Pourquoi ngrok ?

Wise et GoCardless envoient des notifications webhook depuis leurs serveurs externes vers Clenzy à chaque changement d'état de virement. Votre localhost:8084 n'est pas joignable depuis Internet — ngrok crée un tunnel HTTPS sécurisé qui relaie les requêtes vers votre machine.

Installation

brew install ngrok/ngrok/ngrok

Authentification

  1. Créer un compte gratuit sur ngrok.com
  2. Copier le token depuis le dashboard ngrok (section Your Authtoken)
  3. Configurer le token localement : 
    ngrok config add-authtoken <votre-token-ngrok>

Lancer le tunnel sur le port backend Clenzy

ngrok http 8084

Une URL publique est affichée :

Forwarding   https://a1b2c3d4.ngrok-free.app -> http://localhost:8084

C'est cette URL que vous allez configurer dans Wise et dans GoCardless comme URL de webhook.

Vérification

curl https://a1b2c3d4.ngrok-free.app/api/webhooks/payouts/wise/transfers-state-change

Doit retourner 401 Missing signature — c'est attendu : confirme que le tunnel route correctement.

URL ngrok éphémère

Sur le plan gratuit, l'URL ngrok change à chaque relance. Si vous redémarrez ngrok :

  • Mettre à jour l'URL de webhook dans Wise (Dashboard → Settings → Webhooks)
  • Mettre à jour l'URL dans GoCardless (Dashboard → Developers → Webhooks)
  • Mettre à jour gocardless.debtor-account-id n'est pas concerné — il reste fixe

Le plan payant ngrok ($8/mois) permet d'avoir un sous-domaine fixe ce qui simplifie nettement le workflow.

3. Wise Business — Sandbox immédiat Disponible

Wise propose un environnement sandbox auto-service, gratuit et sans KYB. Vous obtenez un compte de test en quelques minutes avec un solde fictif et un Profile ID utilisable pour tester l'intégration complète.

3.1 — Création du compte sandbox

  1. Aller sur https://sandbox.transferwise.tech/register
  2. Choisir le type Business (pas Personal — l'API Business est différente)
  3. Renseigner email + mot de passe + nom d'entreprise (libre, aucune vérification)
  4. Confirmer l'email
  5. Se connecter au dashboard sandbox

3.2 — Récupération des credentials

CredentialOù le trouverChamp Clenzy correspondant
API Token (secret) Dashboard → Settings → API tokens → bouton "Create new token" avec scope Full access wise.api-token
Profile ID (numérique) Dashboard → Account → en haut, sous le nom du business. Format : un long entier (ex: 12345678) wise.profile-id

Format du token : 11111111-2222-3333-4444-555555555555 (UUID).

3.3 — Configuration du webhook

  1. Dashboard Wise → Settings → Webhooks
  2. Cliquer sur "Subscribe to a webhook"
  3. Renseigner l'URL :
    https://<votre-tunnel-ngrok>/api/webhooks/payouts/wise/transfers-state-change
  4. Sélectionner l'événement transfers#state-change
  5. Wise affiche une clé publique RSA — copier le contenu PEM (commence par -----BEGIN PUBLIC KEY-----)
  6. Sauvegarder

3.4 — Variables d'environnement Clenzy

Dans server/src/main/resources/application.yml (ou via variables d'env) :

wise:
  api-token: ${WISE_API_TOKEN:11111111-2222-3333-4444-555555555555}
  profile-id: ${WISE_PROFILE_ID:12345678}
  sandbox: true                         # false en production
  public-key: |                          # Clé publique RSA pour vérifier les webhooks
    -----BEGIN PUBLIC KEY-----
    MIIBIjANBgkqhkiG9w0BAQEFAAOCAQ8AMIIBCgKCAQEAyMEdY1aR...
    -----END PUBLIC KEY-----
Wise unique pour Clenzy entier, pas par-tenant

Contrairement à PayTabs/CMI (paiements entrants, credentials par-org), Wise est un compte unique pour toute la plateforme Clenzy. Toutes les organisations utilisent le même compte Wise Business comme source pour les payouts. La configuration est globale, pas dans la BDD multi-tenant.

3.5 — Solde du compte sandbox

Pour pouvoir effectuer des virements, le compte Wise sandbox doit avoir un solde. Top-up :

  1. Dashboard Wise → Balances
  2. Sélectionner EUR (ou la devise source de Clenzy)
  3. Cliquer "Add money" → choisir Bank transfer (sandbox)
  4. Le sandbox crédite automatiquement (~10 secondes)
  5. Vérifier le solde EUR : doit être > 0 pour que les virements partent

3.6 — Couverture géographique

Wise sandbox supporte tous les pays de production. Pour Clenzy, devises clés :

DevisePaysDélai virement sandbox
EUR → MADvers MarocQuasi-instantané en sandbox
EUR → SARvers Arabie SaouditeQuasi-instantané en sandbox
EUR → EURZone SEPAQuasi-instantané en sandbox
EUR → USDUSAQuasi-instantané en sandbox

4. GoCardless PIS — Sandbox immédiat Disponible EU PSD2

GoCardless Bank Account Data inclut un service Payment Initiation (PIS) qui permet d'initier des virements SEPA Credit Transfer depuis le compte Clenzy bancaire sans uploader de XML pain.001. Le sandbox est gratuit et inclut une banque virtuelle de test (SANDBOXFINANCE) qui simule le flow complet de SCA.

4.1 — Création du compte sandbox

  1. Aller sur https://bankaccountdata.gocardless.com/register/
  2. Renseigner email + mot de passe + nom de la société
  3. Confirmer l'email
  4. Dashboard sandbox accessible immédiatement

4.2 — Récupération des credentials

CredentialOù le trouverChamp Clenzy correspondant
Secret ID Dashboard → User secrets → "Generate new secret" gocardless.secret-id
Secret Key (secret) Affiché une seule fois lors de la génération — copier immédiatement gocardless.secret-key

4.3 — Setup initial : authentification de votre compte bancaire Clenzy

Pour pouvoir initier des virements PIS, GoCardless doit avoir un consent valide pour votre compte bancaire Clenzy (le débiteur). C'est un SCA one-time à faire en tant qu'admin.

  1. Une fois Clenzy démarré, ouvrir Settings → Reversements → Mes reversements (en mode admin/host)
  2. Cliquer sur "Choisir une autre méthode"
  3. Sélectionner "Open Banking PIS"
  4. Choisir SANDBOXFINANCE_SFIN0000 dans le menu déroulant (la banque de test GoCardless)
  5. Cliquer sur "Connecter ma banque"
  6. Le browser est redirigé vers le portail SCA sandbox de GoCardless
  7. Choisir un compte fictif (ex: "Current Account")
  8. Authoriser (l'authentification est simulée — aucun mot de passe requis en sandbox)
  9. Browser redirigé vers Clenzy avec ?openbanking=callback
  10. Le frontend appelle automatiquement finalizeOpenBankingCallback → consent activé pour 90 jours

4.4 — Récupération de l'ID du compte débiteur

Une fois le consent validé, GoCardless vous a fourni un account_id qui identifie votre compte bancaire pour les futurs virements. Cet ID est nécessaire dans la config Clenzy.

  1. Dashboard GoCardless → Account & transactions → Accounts
  2. Trouver le compte qu'on vient d'autoriser
  3. Copier l'Account ID (UUID)

4.5 — Configuration du webhook

  1. Dashboard GoCardless → Developers → Webhook endpoints
  2. Cliquer "Create new endpoint"
  3. URL : https://<votre-tunnel-ngrok>/api/webhooks/payouts/gocardless/events
  4. Sélectionner les événements :
    • payments.paid
    • payments.confirmed
    • payments.failed
    • payments.cancelled
    • payments.customer_approval_denied
  5. GoCardless génère un webhook secret — copier immédiatement

4.6 — Variables d'environnement Clenzy

gocardless:
  secret-id: ${GOCARDLESS_SECRET_ID}
  secret-key: ${GOCARDLESS_SECRET_KEY}
  debtor-account-id: ${GOCARDLESS_DEBTOR_UUID}    # UUID du compte Clenzy
  webhook-secret: ${GOCARDLESS_WEBHOOK_SECRET}

4.7 — Banques disponibles en sandbox

GoCardless expose ~2400 banques européennes en production. En sandbox, on a une banque virtuelle de test :

Institution IDNomPaysUsage
SANDBOXFINANCE_SFIN0000Sandbox FinanceFR/EUTests généraux, SCA simulé
BNP_PARIBAS_BNPAFRPPBNP ParibasFRProd — KYB requis
SOCIETE_GENERALE_SOGEFRPPSociété GénéraleFRProd — KYB requis

6. Scénarios de test end-to-end

6.1 — Scénario Wise complet (EUR → MAD, propriétaire au Maroc)

Préparation : Wise sandbox configuré, solde EUR > 1000€, webhook actif, ngrok actif, propriétaire fictif créé dans Clenzy avec IBAN marocain.

  1. SUPER_ADMIN ouvre Settings → Reversements → table des propriétaires
  2. Clic sur l'icône engrenage du propriétaire marocain → modale "Méthode de reversement"
  3. Sélectionner "Wise Business" dans les options
  4. Renseigner l'IBAN marocain (ex: MA64011519000001205000534921) + Titulaire
  5. Cliquer "Enregistrer"
  6. Générer un payout via le scheduler ou manuellement depuis Facturation → Reversements
  7. Approuver le payout (status passe à APPROVED)
  8. Exécuter le payout (clic icône "Exécuter") :
    • Backend : WisePayoutExecutor.execute()
    • 1er appel : POST /v1/accounts crée le recipient Wise (idempotent côté Wise)
    • 2e appel : POST /v3/profiles/<id>/quotes obtient un quote EUR → MAD
    • 3e appel : POST /v1/transfers crée le transfer avec idempotence UUID
    • 4e appel : POST /v3/profiles/<id>/transfers/<tid>/payments funde depuis le solde
  9. Payout passe en PROCESSING avec paymentReference = "WISE:<transferId>"
  10. Wise traite le virement (quasi-instantané en sandbox)
  11. Webhook transfers#state-change reçu avec current_state = outgoing_payment_sent
  12. Vérification signature RSA-SHA256 avec la clé publique Wise
  13. Lookup payout par paymentReference → mark PAID + notify success
  14. Dashboard Wise : transfer visible dans l'historique

6.2 — Scénario Open Banking complet (consent + virement)

  1. Propriétaire (ou admin) ouvre Mes reversements
  2. Clic "Choisir une autre méthode" → modale
  3. Sélectionner "Open Banking PIS" → dropdown se charge depuis GoCardless
  4. Choisir SANDBOXFINANCE_SFIN0000
  5. Clic "Connecter ma banque" :
    • Frontend : POST /api/owner-payout-config/me/openbanking/init
    • Backend : GoCardlessPisClient.createRequisition() → URL SCA + requisitionId stocké en BDD
    • Frontend : window.location.href = redirectUrl
  6. Portail SANDBOXFINANCE : sélection du compte fictif + auth simulée
  7. Redirect vers Clenzy : app.clenzy.fr/settings?tab=reversements&openbanking=callback
  8. useEffect dans MyPayoutSettings détecte le query param → appel finalizeOpenBankingCallback
  9. Backend : vérifie le consent valide chez GoCardless → marque verified=true + payoutMethod=OPEN_BANKING + consentExpiresAt = now() + 90 days
  10. Snackbar "Méthode de reversement mise à jour"
  11. Bannière OpenBankingConsentBanner affichée avec consent actif
  12. Maintenant, à chaque payout APPROVED → OpenBankingPayoutExecutor.execute() initie un payment GoCardless
  13. Webhook payments.paid reçu → mark PAID

6.3 — Tester l'expiration du consent (forcer date passée)

Pour valider la bannière proactive, on peut forcer une date d'expiration passée en BDD :

docker exec -it clenzy-postgres-dev psql -U clenzy -d clenzy -c \
  "UPDATE owner_payout_config
   SET open_banking_consent_expires_at = NOW() - INTERVAL '1 day'
   WHERE owner_id = <OWNER_ID>;"

Recharger la page Mes reversements → bannière rouge "Consent bancaire expiré" avec bouton "Reconnecter ma banque".

Pour tester l'avertissement J-7 :

UPDATE owner_payout_config
SET open_banking_consent_expires_at = NOW() + INTERVAL '3 days'
WHERE owner_id = <OWNER_ID>;

→ bannière jaune "Consent bancaire à renouveler bientôt".

6.4 — Tester un refund Wise

L'API Wise ne supporte pas les refunds directs — un payout envoyé ne peut pas être annulé une fois outgoing_payment_sent. Si le transfer est encore en état incoming_payment_waiting, on peut le cancel via dashboard.

Pour Clenzy : WisePayoutExecutor.refundPayment n'est pas implémenté (hors scope payouts). Si un refund est nécessaire, il faut le faire manuellement via dashboard Wise + appeler markAsFailed côté Clenzy.

Via curl direct (utile si on doute de la validité côté GoCardless) :

# 1. Obtenir un token
TOKEN=$(curl -s -X POST https://bankaccountdata.gocardless.com/api/v2/token/new/ \
  -H "Content-Type: application/json" \
  -d '{"secret_id":"'$GOCARDLESS_SECRET_ID'","secret_key":"'$GOCARDLESS_SECRET_KEY'"}' \
  | jq -r .access)

# 2. Lookup une requisition
curl -s https://bankaccountdata.gocardless.com/api/v2/requisitions/<REQ_ID>/ \
  -H "Authorization: Bearer $TOKEN" | jq .

# Status attendu : "LN" (Linked) pour un consent valide

9. Annexes — webhooks & structure des payloads

9.1 — Webhook Wise (transfers#state-change)

URL Clenzy : POST /api/webhooks/payouts/wise/transfers-state-change

Header signature : X-Signature-SHA256: <base64-encoded RSA signature>

Body (extrait) :

{
  "data": {
    "resource": {
      "id": 123456789,
      "type": "transfer",
      "profile_id": 12345678,
      "account_id": 87654321
    },
    "current_state": "outgoing_payment_sent",
    "previous_state": "processing",
    "occurred_at": "2026-05-23T10:42:00Z"
  },
  "subscription_id": "subscription-uuid",
  "event_type": "transfers#state-change",
  "schema_version": "2.0.0",
  "sent_at": "2026-05-23T10:42:01Z"
}

États possibles :

  • processing : transfer reçu côté Wise, en attente d'execution
  • funds_converted : conversion devise effectuée
  • outgoing_payment_sent : virement émis → Clenzy mark PAID
  • funds_refunded, cancelled, charged_back, bounced_back → Clenzy mark FAILED

9.2 — Webhook GoCardless (payments.*)

URL Clenzy : POST /api/webhooks/payouts/gocardless/events

Header signature : Webhook-Signature: <hex HMAC-SHA256>

Body (extrait) :

{
  "events": [
    {
      "id": "EV01XXX",
      "created_at": "2026-05-23T10:42:00Z",
      "resource_type": "payments",
      "action": "paid",
      "links": {
        "payment": "PM01XXX",
        "creditor": "CR01XXX"
      },
      "details": {
        "origin": "bank",
        "cause": "payment_paid",
        "scheme": "sepa_credit_transfer"
      }
    }
  ]
}

Actions traitées par Clenzy :

  • paid, confirmed : succès → mark PAID
  • failed, cancelled, customer_approval_denied : échec → mark FAILED

9.3 — Vérification de signature Wise (RSA-SHA256)

Signature signer = Signature.getInstance("SHA256withRSA");
signer.initVerify(wisePublicKey);
signer.update(rawPayloadBody.getBytes(StandardCharsets.UTF_8));
byte[] receivedSig = Base64.getDecoder().decode(headerSignature);
boolean valid = signer.verify(receivedSig);

9.4 — Vérification de signature GoCardless (HMAC-SHA256)

Mac mac = Mac.getInstance("HmacSHA256");
mac.init(new SecretKeySpec(webhookSecret.getBytes(UTF_8), "HmacSHA256"));
byte[] computed = mac.doFinal(rawPayloadBody.getBytes(UTF_8));
String computedHex = HexFormat.of().formatHex(computed);
boolean valid = constantTimeEquals(computedHex, headerSignature);
